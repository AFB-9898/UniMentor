\section{Guía de Exposición}

\subsection{Paso 1: ¿Qué es UniMentor?}
\nota{Empezar con una visión general. No entrar en detalles técnicos todavía.}

\begin{itemize}
  \item ``UniMentor es un marketplace de mentorías universitarias que conecta estudiantes con egresados para orientación profesional.''
  \item Pertenece a la línea de producto \textbf{Plataformas Educativas (EdTech)}.
  \item El MVP cubre: búsqueda de mentores, perfiles con calificación, sistema de rating (1--5 estrellas), y gestión de sesiones.
\end{itemize}

\subsection{Paso 2: Arquitectura}
\nota{Mostrar el diagrama de arquitectura.}

\begin{itemize}
  \item Usamos \textbf{Atomic Design}: atoms $\to$ molecules $\to$ organisms $\to$ screens.
  \item El frontend es React + TypeScript con Vite.
  \item La navegación usa \textbf{React Router} con rutas para Home, agendamiento y lista de sesiones.
  \item El estado de las sesiones se gestiona con \textbf{Context API} para compartir datos entre pantallas.
  \item El backend es \textbf{InsForge} (Backend as a Service) que nos da base de datos PostgreSQL, autenticación y almacenamiento de archivos.
  \item Los componentes reutilizables están en la carpeta \texttt{src/components/}.
\end{itemize}

\pregunta{¿Por qué Atomic Design y no otra arquitectura?}
\respuesta{Atomic Design nos permite construir componentes desde lo más pequeño (atoms) hasta lo más complejo (organisms), asegurando que cada pieza sea reutilizable. Esto es ideal para una línea de producto porque los mismos componentes pueden usarse en diferentes productos de la misma familia.}

\subsection{Paso 3: Componentes Reutilizables}
\nota{Mostrar cada componente en el código. Tener los archivos abiertos en el editor.}

\subsubsection{RatingStars (Atom)}
\begin{itemize}
  \item Componente de calificación por estrellas (1--5).
  \item \textbf{Props:} \texttt{value}, \texttt{interactive}, \texttt{onChange}, \texttt{size}.
  \item \textbf{Nivel:} Atom --- no depende de ningún otro componente.
  \item \textbf{Dónde se reusa:} Perfil de mentor (mostrar rating), feedback post-sesión (calificar), reseñas.
\end{itemize}

\pregunta{¿Por qué es un Atom y no un Molecule?}
\respuesta{Porque no combina otros componentes. Es una unidad visual y funcional primitiva. No depende de nada más que de sus props.}

\subsubsection{SearchFilterBar (Molecule)}
\begin{itemize}
  \item Barra de búsqueda con filtros dinámicos configurables.
  \item \textbf{Props:} \texttt{fields}, \texttt{onFilter}, \texttt{placeholder}.
  \item \textbf{Nivel:} Molecule --- combina input + selects con estado local.
  \item \textbf{Dónde se reusa:} Buscar mentores, filtrar estudiantes, historial de sesiones.
\end{itemize}

\subsubsection{UserProfileCard (Organism)}
\begin{itemize}
  \item Tarjeta de perfil que integra \texttt{RatingStars} y muestra datos del usuario.
  \item \textbf{Props:} \texttt{user}, \texttt{variant}, \texttt{actions}.
  \item \textbf{Nivel:} Organism --- orquesta componentes más pequeños y conoce el dominio.
  \item \textbf{Dónde se reusa:} Resultados de búsqueda, perfil detallado, dashboard.
\end{itemize}

\pregunta{¿Cuál es la diferencia práctica entre Molecule y Organism?}
\respuesta{Un Molecule es una combinación de atoms que sigue siendo independiente del dominio (como una barra de búsqueda). Un Organism ya conoce los datos del negocio (como un perfil de usuario) y orquesta varios componentes.}

\subsection{Paso 4: Refactorización}
\nota{Mostrar el antes y después en el código.}

\subsubsection{Extract Method}
\begin{itemize}
  \item \textbf{Problema:} Código de formateo de fechas duplicado en varios componentes.
  \item \textbf{Solución:} Extraer a \texttt{src/utils/formatDate.ts} --- una sola función reutilizable.
  \item \textbf{Beneficio:} Un solo punto de cambio, mejor legibilidad.
\end{itemize}

\pregunta{¿Qué principio estabas violando antes?}
\respuesta{DRY (Don't Repeat Yourself). Teníamos el mismo código de formateo de fecha en al menos dos componentes.}

\subsubsection{Replace Magic String}
\begin{itemize}
  \item \textbf{Problema:} Strings literales como \texttt{"pending"}, \texttt{"confirmed"} sin tipo.
  \item \textbf{Solución:} Crear un tipo \texttt{SessionStatus} en \texttt{src/types/index.ts}.
  \item \textbf{Beneficio:} TypeScript valida en compilación, autocompletado en el editor.
\end{itemize}

\pregunta{¿Por qué es importante esto?}
\respuesta{Si escribís mal un string como \texttt{"confirmed"} en lugar de \texttt{"confirmed"}, JavaScript no te avisa hasta que explota en runtime. Con el tipo centralizado, TypeScript te marca el error al escribir el código.}

\subsection{Paso 5: Testing}
\nota{Mencionar que todo está testeado.}

\begin{itemize}
  \item \textbf{Total:} 27 tests unitarios.
  \item \textbf{Stack:} Vitest + React Testing Library + jsdom.
  \item \textbf{Cobertura:} RatingStars (8 tests), SearchFilterBar (5 tests), UserProfileCard (9 tests), formatDate (5 tests).
  \item Para ejecutar: \texttt{npm run test}
\end{itemize}

\pregunta{¿Probaste algo más que renderizado?}
\respuesta{Sí, probamos interacción: clics en estrellas (modo interactivo), escritura en filtros, envío de formulario de búsqueda, y casos borde como fechas inválidas.}

\section{Posibles Preguntas y Respuestas}

\begin{enumerate}

  \item \pregunta{¿Cuál es la línea de producto y por qué elegiste EdTech?}
  \respuesta{Plataformas Educativas (EdTech). La elegí porque es un área con alta demanda de software y donde los componentes reutilizables tienen sentido: perfiles de usuario, sistemas de rating, buscadores, etc., aparecen en muchos productos educativos.}

  \item \pregunta{¿Qué es una línea de producto de software?}
  \respuesta{Es un conjunto de sistemas que comparten una base común (activos reutilizables) y se adaptan a necesidades específicas. En nuestro caso, la línea EdTech comparte componentes como RatingStars, SearchFilterBar y UserProfileCard, y podemos crear diferentes productos educativos a partir de ellos.}

  \item \pregunta{¿Cuántos componentes reutilizables implementaste?}
  \respuesta{Tres: RatingStars (atom), SearchFilterBar (molecule) y UserProfileCard (organism). Cada uno en su nivel de Atomic Design.}

  \item \pregunta{¿Qué técnicas de refactorización aplicaste?}
  \respuesta{Dos: Extract Method (para el formateo de fechas) y Replace Magic String (para los estados de sesión).}

  \item \pregunta{¿Dónde está el código?}
  \respuesta{En GitHub: \url{https://github.com/AFB-9898/UniMentor}. El repo incluye todo el código fuente, tests, documentación y configuraciones.}

  \item \pregunta{¿Usaste TypeScript? ¿Por qué?}
  \respuesta{Sí. TypeScript nos da tipado estático que previene errores en compilación, mejora el autocompletado y hace que el código sea más mantenible. Especialmente útil para proyectos con componentes reutilizables donde las interfaces deben ser claras.}

  \item \pregunta{¿Los tests pasan todos?}
  \respuesta{Sí, 27 tests pasando. Se puede verificar ejecutando \texttt{npm run test} en el proyecto.}

  \item \pregunta{¿Cómo aseguraste alta cohesión y bajo acoplamiento?}
  \respuesta{Cada componente tiene una responsabilidad clara (alta cohesión) y se comunica solo a través de sus props (bajo acoplamiento). Por ejemplo, RatingStars no sabe nada de usuarios ni sesiones, solo recibe un valor y opcionalmente un callback.}

  \item \pregunta{¿Implementaste algo adicional fuera de lo pedido?}
  \respuesta{Sí. Implementé un flujo completo de agendamiento con navegación entre pantallas usando React Router: desde la home se selecciona un mentor, se completa un formulario con validación, y se redirige a una pantalla ``Mis Sesiones'' que muestra todo lo agendado con su estado. Además usé Context API de React para compartir el estado de las sesiones entre componentes sin necesidad de un backend.}

\end{enumerate}

\section{Notas Finales}
\nota{Algunos consejos para la defensa:}

\begin{itemize}
  \item Si no sabés una respuesta, no inventes. Decí ``No estoy seguro, pero lo que sé es que...'' y contá lo que sí sabés.
  \item Mostrar el código funcionando vale más que cualquier explicación teórica.
  \item La docente valora que entiendas \textbf{por qué} hiciste cada cosa, no solo que la hiciste.
  \item Si te pide cambiar algo, anotalo y decí ``Lo voy a corregir, gracias por la observación.'' No te pongas a la defensiva.
  \item \textbf{Ejemplo} de buena actitud: ``Usé Atomic Design porque...'' en lugar de ``Usé Atomic Design porque me dijeron que lo use''.
\end{itemize}
